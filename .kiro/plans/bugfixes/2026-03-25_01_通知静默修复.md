# Bug 修复方案：通知静默（无震动、无声音、无横幅弹出）

## Requirements

### R1: 通知必须有声音、震动、横幅弹出
- 描述：任务提醒通知必须以 heads-up 横幅方式弹出，伴随震动和声音。息屏时应亮屏。
- 验收标准：
  - [ ] 点击侧边栏"测试通知"按钮，通知以横幅弹出，有震动有声音
  - [ ] 设置 1 分钟后的任务提醒，到时间后通知横幅弹出
  - [ ] 息屏状态下收到通知时亮屏
  - [ ] dumpsys notification 中 mBuzzBeepBlinkCode 不为 0

## Technical Design

### 根因分析

dumpsys 关键证据：
- `flags=FSI_REQUESTED_BUT_DENIED` — fullScreenIntent 被系统拒绝
- `mBuzzBeepBlinkCode=0` — 系统决定不播放任何提醒效果
- `vibrate=null sound=null defaults=0` — 通知本身的震动/声音字段为空
- 但渠道层面 `mVibrationEnabled=true, mVibrationPattern=[0,500,200,500]` 是正确的

三个叠加原因：

1. **Builder 上的 setVibrate/setSound 在 API 26+ 无效且干扰系统判断**
   - Android 8+ 通知的震动/声音完全由 NotificationChannel 控制
   - Builder 上设了 setVibrate/setSound 后，系统可能将通知本身的 vibrate/sound 清零
   - 对比 Tasks.org：Builder 上完全不设 setVibrate/setSound/setDefaults

2. **Channel 的 sound 用了错误的 URI**
   - 代码用 `Settings.System.DEFAULT_NOTIFICATION_URI`
   - 应该用 `RingtoneManager.getDefaultUri(RingtoneManager.TYPE_NOTIFICATION)`

3. **FSI_REQUESTED_BUT_DENIED 影响 heads-up 显示**
   - Android 14+ USE_FULL_SCREEN_INTENT 需要用户手动授权
   - 被拒绝后可能影响 heads-up notification 的显示
   - 解法：移除 USE_FULL_SCREEN_INTENT 权限声明

### 修改概览

参考 Tasks.org（最知名开源 Android Todo App）的通知实现：
- Builder 上不设 setVibrate/setSound/setDefaults，让 Channel 完全控制
- Channel 用 RingtoneManager 获取默认通知铃声
- 不依赖 fullScreenIntent
- 用 NotificationManagerCompat 替代 NotificationManager

### 详细设计

**注意：之前的开发 agent 已经将修复代码写入了以下文件，但尚未经过审查验证。请先审查再构建。**

已修改的文件：
- `android/app/src/main/kotlin/com/mingh/todolist/MainActivity.kt` — 已重写
- `android/app/src/main/AndroidManifest.xml` — 已移除 USE_FULL_SCREEN_INTENT

关键变更点（需要审查确认）：

1. 新渠道 ID `todo_alarm_v4`，清理所有旧渠道
2. Channel 用 `RingtoneManager.getDefaultUri(TYPE_NOTIFICATION)` + `AudioAttributes(USAGE_NOTIFICATION_EVENT)`
3. Builder 上只设 `setPriority(HIGH)` + `setCategory(REMINDER)` + `setOnlyAlertOnce(false)` + `setVisibility(PUBLIC)`
4. 不设 setVibrate/setSound/setDefaults/setFullScreenIntent
5. 用 `NotificationManagerCompat.from(context).notify()` 发送
6. `ensureChannel` 只执行一次（`channelCreated` 标记）
7. 移除 `USE_FULL_SCREEN_INTENT` 权限

### 回归风险

- 渠道 ID 变更后，用户之前安装的版本的旧渠道会被删除，通知设置重置
- 小米设备可能仍需用户手动开启"横幅通知"权限（代码无法绕过 OEM 限制）

### Task 1: 审查已修改的 MainActivity.kt 和 AndroidManifest.xml
- 目标：确认修复代码正确
- 涉及文件：android/app/src/main/kotlin/com/mingh/todolist/MainActivity.kt, android/app/src/main/AndroidManifest.xml

- 验证方式：quick-review agent 审查
- 测试要点：渠道 ID 一致性、Builder 无 setVibrate/setSound、ensureChannel 幂等性

### Task 2: 构建并全新安装测试
- 目标：验证通知横幅弹出、震动、声音
- 步骤：
  1. `flutter clean && flutter build apk --release`
  2. 用户手动卸载旧版本（确保旧渠道清除）
  3. `adb install` 全新安装
  4. 打开 App，允许通知权限
  5. 侧边栏点"测试通知"
  6. `adb shell dumpsys notification | grep -A10 todo_alarm_v4` 检查 mBuzzBeepBlinkCode 是否不为 0
- 测试要点：横幅弹出、震动、声音、dumpsys 验证

### Task 3: 如果 Task 2 仍然静默，检查小米特定设置
- 目标：排除小米 OEM 限制
- 步骤：
  1. `adb shell dumpsys notification | grep -B2 -A15 todo_alarm_v4` 查看完整通知记录
  2. 检查 `mBuzzBeepBlinkCode` 值
  3. 如果仍为 0，检查小米"横幅通知"设置：设置 → 应用设置 → 应用管理 → Todo → 通知管理 → 任务提醒 → 确认"横幅通知"已开启
  4. 如果"横幅通知"关闭，用 `adb shell am start -a android.settings.CHANNEL_NOTIFICATION_SETTINGS --es android.provider.extra.APP_PACKAGE com.mingh.todolist --es android.provider.extra.CHANNEL_ID todo_alarm_v4` 直接跳转到渠道设置页
