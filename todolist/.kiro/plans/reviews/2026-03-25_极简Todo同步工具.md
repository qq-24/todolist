## 🔴 严重问题（必须修复）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 问题 1：「完成」按钮名不副实 — 点了不完成任务
  │ 模块：Android 原生层
  │ 触发方式：提醒弹出 → 用户点击悬浮窗「完成」按钮
  │ 用户感受：以为任务已完成，下次打开 App 发现任务仍未完成
  │ 根因：MainActivity.kt ReminderOverlay 的「完成」按钮只调用 ReminderAlert.stop() + dismiss()，没有通过 MethodChannel 通知 Flutter 侧标记任务完成
  ┃ 建议修复：通过 MethodChannel 回调 Flutter 侧 toggleComplete(id)；或将按钮文案改为「知道了」

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 问题 2：悬浮窗权限未授权时铃声无法停止
  │ 模块：Android 原生层
  │ 触发方式：用户未授予悬浮窗权限 → 闹钟触发
  │ 用户感受：铃声一直响，没有任何 UI 可以关闭，只能杀掉 App
  │ 根因：AlarmReceiver.onReceive() 先启动铃声/震动，再调用 ReminderOverlay.show()。show() 检测到无权限直接 return，但铃声已经在播放
  ┃ 建议修复：show() 返回 boolean，返回 false 时设置 30 秒超时自动 stop()；或给备用通知添加「停止」action

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 问题 3：「稍后提醒」的闹钟 ID 不受 Flutter 侧管理
  │ 模块：Android 原生层
  │ 触发方式：点击「稍后提醒」→ 5 分钟内在 App 中删除该任务
  │ 用户感受：已删除的任务 5 分钟后仍然弹出提醒
  │ 根因：稍后提醒用 id+100000 设置新闹钟，但 Flutter 侧 _cancelAllForTodo 只取消原始 ID 的闹钟
  ┃ 建议修复：在 Dart 侧 cancelReminder 时同时取消 id+100000 的闹钟；或改用可追踪的 ID 方案

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 问题 4：震动时长过短且不重复（4.4 秒一次性）
  │ 模块：Android 原生层
  │ 触发方式：任何提醒触发
  │ 用户感受：手机在口袋里或远处时，4.4 秒震动很容易错过
  │ 根因：VibrationEffect.createWaveform(pattern, -1) 的 repeat=-1 表示不循环
  ┃ 建议修复：repeat 参数改为 0（循环），配合 60 秒超时自动停止

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 问题 5：铃声无自动超时停止
  │ 模块：Android 原生层
  │ 触发方式：提醒触发后用户长时间不交互
  │ 用户感受：铃声一直响直到电池耗尽或系统杀进程
  │ 根因：Ringtone.play() 后没有任何超时停止机制
  ┃ 建议修复：添加 60 秒 Handler.postDelayed 自动调用 ReminderAlert.stop()

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 问题 6：vibrate_in_dnd 设置存储不一致 — Flutter 和原生读的不是同一个文件
  │ 模块：Android 原生层 + 设置
  │ 触发方式：用户在侧边栏修改「勿扰时也震动」开关
  │ 用户感受：开关改了但提醒时行为没变
  │ 根因：Flutter SharedPreferences 存在 FlutterSharedPreferences.xml（key 带 flutter. 前缀），原生侧用 getSharedPreferences("todo_prefs") 读的是另一个文件
  ┃ 建议修复：原生侧改为读取 FlutterSharedPreferences.xml 并用 flutter.vibrate_in_dnd 作为 key；或通过 MethodChannel 传递设置值

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 问题 7：设备重启后所有闹钟丢失，无恢复机制
  │ 模块：Android 原生层
  │ 触发方式：用户设置了提醒后重启手机
  │ 用户感受：重启后所有提醒静默消失，任务到期无通知
  │ 根因：AndroidManifest.xml 未注册 RECEIVE_BOOT_COMPLETED 和 BootReceiver
  ┃ 建议修复：注册 BootReceiver，开机时通过 Flutter 引擎或本地存储重新调度所有闹钟

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 问题 8：多个提醒同时触发时只显示最后一个
  │ 模块：Android 原生层
  │ 触发方式：两个任务的提醒时间相差几秒内
  │ 用户感受：第一个提醒的悬浮窗被第二个替换，铃声/震动被重置，第一个任务信息丢失
  │ 根因：ReminderAlert 和 ReminderOverlay 都是 object 单例，start()/show() 开头先 stop()/dismiss()
  ┃ 建议修复：使用队列机制，关闭当前悬浮窗后自动显示下一个

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 问题 9：冲突解决失败后冲突状态被清除，用户无法重试
  │ 模块：同步逻辑
  │ 触发方式：冲突弹窗 → 选择「保留本地」→ push 失败（网络问题）
  │ 用户感受：弹窗消失，只剩红色错误图标，不知道选择是否生效，无法重试
  │ 根因：todo_provider.dart resolveConflict() 中 push 失败时清空 _conflictRemoteData
  ┃ 建议修复：push 失败时保留冲突数据，让用户可以重试

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 问题 10：_save() 未 await 持久化，保存后立即关闭面板可能丢数据
  │ 模块：UI 层
  │ 触发方式：用户保存任务后立即杀掉 App
  │ 用户感受：以为保存成功了但重新打开发现任务不见了
  │ 根因：todo_edit_sheet.dart _save() 是同步方法，调用异步的 addTodo/updateTodo 后立即 pop()
  ┃ 建议修复：_save 改为 async，await provider.addTodo()/updateTodo() 完成后再 pop()

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 问题 11：用户可设置过去的截止时间，提醒被静默跳过
  │ 模块：UI 层
  │ 触发方式：编辑旧任务，设置已过去的截止时间，开启提醒
  │ 用户感受：设了提醒但永远收不到，完全不知道为什么
  │ 根因：todo_edit_sheet.dart firstDate 允许昨天，时间选择器无下限；reminder_service.dart 静默跳过过期时间
  ┃ 建议修复：保存时校验 deadline 是否在未来，至少给出警告

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 问题 12：_buildTodoFile() 每次推送都刷新 _localUpdatedAt，导致冗余推送
  │ 模块：同步逻辑
  │ 触发方式：30 秒轮询触发 sync，即使数据没变也会推送
  │ 用户感受：GitHub 仓库产生大量无意义 commit
  │ 根因：todo_provider.dart _buildTodoFile() 每次调用都设 _localUpdatedAt = DateTime.now()，导致本地时间戳总是比远端新
  ┃ 建议修复：推送时使用已有的 _localUpdatedAt 而不是重新生成；或只在实际数据变化时更新时间戳

## 🔵 优化建议

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
▸ 建议 1：移除 3 个未使用的依赖
  │ flutter_local_notifications、android_alarm_manager_plus、timezone 在 lib/ 中无任何 import，增加包体积
  ┃ 预期效果：减小 APK 体积，加快编译

▸ 建议 2：ReminderService 应为单例
  │ TodoProvider 和 AppDrawer 各创建独立实例，Windows 端 Timer 互相独立
  ┃ 预期效果：统一管理提醒状态

▸ 建议 3：GithubSyncService._lastSha 是死代码
  │ sync 流程总是用 fetch 返回的 sha，_lastSha 从未被外部使用
  ┃ 预期效果：减少混淆

▸ 建议 4：本地存储 load() 失败时应备份损坏文件
  │ JSON 解析失败时直接返回 null，损坏数据无法恢复
  ┃ 预期效果：方便排查数据丢失问题

▸ 建议 5：proguard-rules.pro keep 规则过于宽泛
  │ 保留了整个包的所有类和成员，等于禁用混淆
  ┃ 预期效果：缩小 APK 体积

▸ 建议 6：轮询不感知 App 前后台状态
  │ 后台仍在轮询，回到前台没有立即同步
  ┃ 预期效果：节省流量，前台数据更及时

▸ 建议 7：标题和描述无长度限制
  │ 极端情况下超长文本导致数据膨胀
  ┃ 预期效果：防止异常数据

▸ 建议 8：编辑未修改时仍触发保存和同步
  │ 无变化时应直接 pop() 不触发写入
  ┃ 预期效果：减少无意义同步

▸ 建议 9：编辑面板关闭无「放弃编辑？」确认
  │ 用户误触可能丢失已输入内容
  ┃ 预期效果：防止误操作

▸ 建议 10：稍后提醒固定 5 分钟，无法自定义
  │ 用户可能需要不同延迟时长
  ┃ 预期效果：更灵活的提醒体验

▸ 建议 11：悬浮窗权限引导时机不佳
  │ 首次打开就弹权限请求，用户不理解用途
  ┃ 预期效果：在用户首次开启提醒时再请求，提高授权率

▸ 建议 12：Todo.fromJson 中 deadline 字段 as String 可能 cast 失败
  │ 远端数据被篡改时会崩溃
  ┃ 预期效果：更健壮的数据解析

## 预计改进效果
